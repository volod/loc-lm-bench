## Problem Statement

Build a lightweight, production-grade, reproducible internal tool that selects the
best open-weight LLM for a specific language (default Ukrainian) for RAG and text
analysis, running on a single small desktop GPU (validated target: RTX 4060 Ti
16 GB). The output is a defensible model pick for an internal pipeline, not a
public benchmark platform.

The original spec is over-scoped for this goal: 9 benchmark categories, 3 backends
as a comparison axis, 6 agent frameworks, 4 vector stores, security/jailbreak,
MCP/tooling, and a full reproducible leaderboard. That is a multi-month research
platform. It does not answer the actual question faster than the platform takes to
build.

## Demand Evidence

This is an internal tool, so demand is operational rather than market-based. The
decisive evidence is the user's current workflow:

- Today the user picks models from public leaderboards only (HF Open LLM
  Leaderboard, community Ukrainian benchmarks).
- The pain: those leaderboards measure general capability on someone else's tasks,
  in someone else's quantization, with unlimited VRAM. They do not test this
  corpus, this RAG stack, this 16 GB desktop, or this task shape. So the public
  ranking may not transfer, and the user cannot defend "why this model."

The real gap is transfer validity, not infrastructure. Both Claude and an
independent Codex cold read reached this conclusion separately. The product is an
internal decision machine: re-rank models on *my* corpus, *my* task, *my* GPU.

The problem is also shared beyond this user: specific-language / low-resource-
language model selection on constrained hardware is underserved by public
English-centric leaderboards. The same harness generalizes to other languages by
swapping the corpus, gold set, and embedding model.

## Status Quo

"Public leaderboards only." Pick a reputable multilingual model, run it, hope the
public ranking transfers. No measurement on own data, no reproducible record, no
defensible justification, no awareness of whether a better fit exists for the
hardware.

## Target User & Narrowest Wedge

Target user: the engineer (and the internal pipeline, e.g. the parent project) who
must commit to one model for Ukrainian RAG + text analysis and justify the choice.

Narrowest wedge (chosen): RAG + text analysis, scored on the user's own corpus,
across ~6-10 candidate models, on the user's GPU. Score = quality (objective
retrieval + gated LLM-judge generation) plus tokens/sec and does-it-fit-in-VRAM.
Everything else deferred.

## Constraints

- Single small desktop GPU (16 GB class validated). v1 needs only single-GPU fit
  detection on the host. Cross-class (12/24/48 GB) VRAM-aware planning stays in
  Approach B and is not built now.
- Ukrainian is the default target language; the design must not assume English.
- Production-grade: reproducible, recorded, defensible, low-maintenance, must
  survive months without babysitting.
- At most a few dozen models and a few hundred to few thousand eval cases. No
  queues, no Kubernetes, no separate services.

## Premises

1. (Revised by user) Backend is resolved PER MODEL by availability priority:
   vLLM (HF weights) > Ollama (Ollama library) > llama.cpp (GGUF / heavy quant),
   plus hardware fit. Backends are NOT a comparison axis in v1. All three expose
   OpenAI-compatible HTTP, so the eval/RAG/judge code is backend-agnostic; only the
   launcher and the availability resolver are backend-specific. Same-model-across-
   backends comparison is deferred.
2. The LLM-judge is a GATED dependency. It is calibrated against a fixed set of 30
   human-rated answers (a subset of the gold set, spanning at least 3 models and the
   full score range) using the EXACT Ragas judge configuration that will produce the
   scores, not a separate hand-rolled rubric, with Ragas's metric prompts localized
   to Ukrainian. Pass bar (v1 default): Spearman rho >= 0.6 between judge and human
   rankings over the calibration set (0.6 is a moderate-correlation floor; below it the
   judge diverges too far from human judgment to trust, re-tune after the first real
   calibration). Below that bar, or if scores are unstable under
   prompt variants or biased toward fluent-but-wrong answers, the judge is demoted to
   a diagnostic and the objective retrieval + reference-based scores carry the
   ranking. A small 7B local judge is not trustworthy for Ukrainian quality.
3. The Ukrainian gold set is the real work, not the harness: ~50-150 RAG items
   (question + correct chunk labels + reference answer) and ~30-50 text-analysis
   items, drawn from the user's own corpus. Harness code is the easy ~30%;
   defensible Ukrainian ground-truth is the ~70% and the main time cost.
4. The embedding model is validated SEPARATELY and FIRST, then pinned and recorded.
   Retrieval quality is a confound: a weak Ukrainian embedding model silently caps
   every generation model's RAG score and contaminates the comparison. Multilingual
   does not imply good Ukrainian domain recall. Acceptance gate (v1 default): the
   chosen embedding model must reach recall@10 >= 0.8 on the gold set's labeled
   chunks (0.8 means the correct chunk lands in the top 10 for 4 of 5 questions;
   below that, retrieval, not the generation model, is the bottleneck). If no
   candidate clears the bar, pick the best available, record the
   retrieval ceiling in the manifest, and flag that all RAG scores are capped by
   retrieval.
5. Reuse over rebuild: Ragas (RAG metrics + judge orchestration), OpenAI-compatible
   client (single backend interface), one vector store (FAISS for v1), DuckDB +
   Parquet for results, Streamlit for the board. The user writes glue + gold set +
   judge calibration + backend launchers, not eval primitives.
6. Deferred and written down as "later," not built: security/jailbreak, agentic,
   MCP, tooling, the 6 agent frameworks, multi-backend comparison, multi-vector-
   store, the full GPU matrix, quality-per-watt, automated dataset generation.

## Approaches Considered

### Approach A: Thin decision loop (chosen base)
A Typer CLI under `src/llb/` with a small set of commands, one pinned embedding
model + FAISS, gold set as JSONL, launchers only for the backends the first
candidates need, Ragas + objective retrieval scoring with a gated judge, DuckDB +
Parquet storage, Streamlit board. Effort M (CC ~1-2 focused days), risk low. Ships
a defensible pick fast, heavy reuse, small maintenance surface.

### Approach B: Backend-agnostic harness with a real per-model resolver (chosen seams)
Approach A plus a `BackendLauncher` interface (vLLM / Ollama / llama.cpp), an
`AvailabilityResolver` (HF Hub + Ollama library + GGUF availability, priority +
VRAM fit), a hardware detector (NVML/nvidia-smi parse) feeding a planner that
validates model+backend+context+VRAM before each run, full per-run reproducibility
manifests, and psutil/NVML telemetry. Effort L (CC ~1 week), risk medium (vLLM CUDA
build). The real production trajectory.

### Approach C: Promptfoo as the spine (rejected)
Lean on Promptfoo for fan-out, caching, side-by-side UI, and rubric grading; write
the gold set as test cases. Least custom code, fastest UI, but Promptfoo's
test-case-centric model fights reproducibility manifests, hardware telemetry, and a
defensible composite leaderboard score. Likely a throwaway proof you outgrow.

## Recommended Approach

Approach A as v1, built with B's two key seams from day one:

1. A `BackendLauncher` abstraction: start/stop a serving backend for a given model
   + config behind OpenAI-compatible HTTP, PLUS a backend-specific telemetry hook.
   Tokens/sec, requested-vs-served context, and peak VRAM are extracted differently
   per backend even though the chat calls are uniform, so each adapter carries a
   telemetry method; this is not "just a launcher." v1 ships the Ollama launcher
   first to prove the full loop with zero CUDA build. The vLLM launcher is added only
   once a committed candidate model requires HF-only weights (see Open Question 3);
   it is NOT built speculatively, because the from-source vLLM/flash-attn build on
   Ada (sm_89) is the single largest risk and can cost days. The interface is in
   place so the vLLM and llama.cpp launchers and the full `AvailabilityResolver` are
   a clean extension, not a rewrite.
2. A per-run reproducibility manifest for every run: model, backend, backend
   version, quantization, GPU model, driver, CUDA, requested vs served context,
   embedding model + version, gold set version, prompt version, judge config +
   calibration result, and scores. For vLLM, `gpu_memory_utilization` is controlled
   and recorded so peak VRAM is comparable across runs (vLLM pre-reserves a KV-cache
   fraction regardless of model, so raw peak VRAM is otherwise not a clean fit
   signal).

Effort, end to end and honest: the harness code is the small part (~1-2 focused CC
days). The gold set plus judge calibration is the bulk and the real cost. Realistic
time to a defensible first pick is ~1-2 weeks of human effort, dominated by gold-set
curation, not coding. The door stays open to the full Approach B without a teardown.

### v1 build shape (informational, not a task list)

- `src/llb/` package Typer CLI: `prepare-corpus`, `prepare-goldset`,
  `prepare-synthetic-corpus`, `build-index`, `validate-embeddings`, `calibrate-judge`,
  `screen-public`, `optimize-config`, `run-eval`, `show-board`.
- Corpus + gold set under `$DATA_DIR/llb/` per AGENTS.md. Vector index = FAISS.
- Pipeline dependency order: embeddings (validated, pinned) -> index -> generate
  answers -> score. The Assignment's judge calibration runs before any of this using
  throwaway manual retrieval, so it genuinely precedes the harness.
- Scoring has TWO distinct layers (see Eng-review resolutions): (1) RETRIEVAL quality
  (recall@k, MRR) computed by overlap against SOURCE-SPAN gold labels; it validates the
  embedding + retrieval config (Premise 4) and is CONSTANT across generation models under
  pinned retrieval, so it is reported as context, NOT a generation-model ranking axis.
  (2) GENERATION quality ranks models: reference-based answer correctness (exact /
  semantic / structured-label match to gold reference answers) as the objective axis,
  plus Ragas faithfulness/answer-relevance via the gated judge as the subjective axis.
  Text-analysis scored against structured planted labels (objective) plus the gated judge.
- Fit/fail detection (v1, single-GPU): attempt to load and serve the model on the
  host; OOM or launcher failure = fail, recorded with reason; success records peak
  VRAM. No predictive planner in v1 (that is Approach B).
- Ranking function (v1 default): fit/fail is a hard filter. Models are ranked by
  GENERATION quality: reference-based answer correctness (objective) blended with the
  gated judge when calibrated, or reference answer correctness ALONE when the judge is
  demoted (NOT retrieval recall, which is constant per model). Ties broken by tokens/sec,
  then lower peak VRAM. Output is a PARETO table (quality vs speed vs VRAM) first, single
  rank second; every number carries a confidence interval. Blend weights in config (Open
  Question 5). Score computed at the Optuna-selected best (backend + RAG) config for that
  model; a model with no VRAM-feasible config is marked infeasible.
- Resumability: results persist per (model, case) to Parquet as they complete;
  `run-eval` is idempotent and resumable, skipping already-completed cells. Required
  for the "survives months unattended" goal and for OOM/timeout recovery mid-run.
- Storage: the source of truth is an immutable per-run manifest (JSON/YAML) + Parquet
  outputs written under `$DATA_DIR` FIRST. MLflow (local file/SQLite, no server) mirrors
  those for its UI; it is a convenience layer, not the canonical record. DuckDB optional
  over the parquet for ad-hoc SQL. See "v1 Engineering Decisions".
- Board: MLflow UI for deep run inspection/compare (free); a THIN Streamlit page for
  the composite ranking + best-config-per-model view only. See "v1 Engineering
  Decisions".

## Open Questions

1. Which exact embedding model(s) to validate first for Ukrainian (candidates to
   benchmark: multilingual-e5-large, bge-m3, paraphrase-multilingual, any Ukrainian-
   tuned embeddings). Decided empirically in `validate-embeddings`.
2. Which LLM-judge to use, and is it an external API (cost + data-egress
   consideration for the corpus) or the largest local model that fits. Gated by
   calibration either way.
3. The first concrete candidate model list (~6-10) and which need vLLM vs Ollama.
4. Whether text-analysis reference answers exist or must be authored alongside the
   RAG gold set.
5. The v1 default blend weights between retrieval and judge quality, and the
   tie-breaker order (speed, then VRAM-fit). The weights live in config and are
   tunable from day one; the open question is only which defaults to ship.

## Success Criteria

- A single command produces a ranked table of ~6-10 models on the user's Ukrainian
  corpus, with retrieval score, gated judge score, tokens/sec, peak VRAM, and
  fit/fail status, on the user's GPU.
- Every ranked entry traces back to a reproducible run manifest.
- The judge's Ukrainian calibration result is recorded; if it failed, the ranking
  visibly relies on objective scores.
- The user can state and defend "why this model" from the board, replacing
  "public leaderboards only."
- Re-running after a new model is added on an already-supported backend updates the
  board without code changes. (A model that needs a not-yet-implemented backend
  requires that launcher to be added first.)

## Distribution Plan

Internal tool, single machine: no package distribution needed for v1. Delivery is
the git repo + `make venv` (or `uv`) setup + a documented run flow. The one CI/build
concern is the vLLM/flash-attn source build: cap MAX_JOBS via the canonical helper
(confirm or create its real path for this repo first, see Open Question 6). Registry and

checkouts are exported under `$DATA_DIR/wheels/<pkg>_<abi-key>_git<revision>/`; pin
torch/flash-attn to the host CUDA/GPU. This build only happens if a v1 candidate actually
requires vLLM. A Docker
image that can target the host's GPU/CUDA is a later option, not required for v1.

## The Assignment

Before writing any harness code: hand-curate 30 Ukrainian RAG items from your real
corpus (question + correct chunk IDs + reference answer), then personally rate ~30
candidate-model answers across at least 3 models and check whether your chosen
LLM-judge, in the exact Ragas config you will use with prompts localized to
Ukrainian, agrees with your ranking. Use throwaway manual retrieval here (paste the
relevant chunks by hand) so this genuinely precedes the harness. If agreement is
poor (Spearman rho < 0.6), you have proven the judge cannot be trusted before
building anything on top of it. This de-risks the single most fragile premise and
produces the seed of both the gold set and the calibration set you need anyway.

## v1 Engineering Decisions (CEO review, 2026-06-19)

HOLD SCOPE review. The wedge is unchanged. These decisions refine HOW v1 is built, add
a sequential-execution correctness contract, and expand the leaderboard output to
include the optimal per-machine configuration.

### Tool stack (lightweight-first, minimize custom code)
- Serving: Ollama / vLLM / llama.cpp via subprocess + their OpenAI-compatible HTTP. No
  Ray/Triton.
- Client: `openai` client as the single backend interface.
- Discovery: `huggingface_hub` + Ollama API.
- RAG: FAISS in-process + `langchain-text-splitters` + `sentence-transformers`. No full
  LlamaIndex/Haystack, no vector-DB server in v1.
- Scoring: Ragas (RAG metrics + judge); recall@k / MRR custom (~20 lines).
- Config search: Optuna (in-process, persistent SQLite study). No Ray Tune.
- Tracking/telemetry: MLflow in LOCAL file/SQLite mode, NO tracking server, used as a UI
  MIRROR. Canonical record = immutable per-run manifest (JSON/YAML) + Parquet under
  `$DATA_DIR/`, written first; MLflow logs from them. MLflow UI for inspection/compare.
- GPU/host telemetry: pynvml + psutil, logged as MLflow metrics.
- LLM flows: all per-case eval flows are LangGraph graphs built from ~3 DRY templates
  (single-call, map-reduce, multi-hop). The run executor stays plain Python.
- Leaderboard UI: thin Streamlit for the composite ranking + best-config-per-model view
  only (MLflow UI covers deep inspection).
- Orchestration: plain Python sequential executor. No Celery/Redis/K8s/Airflow.
- Heavyweight services explicitly excluded from v1: MLflow tracking server, vector-DB
  server, Ray, DCGM/Prometheus, message queues.

### Sequential execution contract (prevents leaks and biased measurements)
- Each (model, config) runs in its OWN freshly-started backend process; the run
  executor kills it after the full case set.
- Between every run (including every Optuna trial): poll NVML until VRAM returns to
  baseline within a TOLERANCE band (drivers/display processes mean it won't hit the exact
  byte), attributing any residual to the backend PID before judging it leaked. Then a
  CAPPED, configurable cooldown: wait up to a max (e.g. 60-120s) for GPU temp to drop
  below threshold, else proceed and FLAG the run thermally-elevated. Always record
  temp/clocks/power.
- If residual VRAM attributable to the killed backend stays above tolerance: raise
  `VramNotReclaimed` and ABORT the suite loudly (a real leak biases the next run). A
  baseline shift from unrelated desktop/display processes is tolerated, not aborted.
- Record per run against this clean, thermally-comparable baseline: VRAM peak/avg, temp,
  clocks, power (if available), load time, tokens/sec.

### Config optimization (Optuna, bounded two-stage)
- Gold set is partitioned into DISJOINT splits: judge-calibration, tuning (the Optuna
  proxy), and final-eval. Tuning never touches final-eval, so config search cannot leak
  into the leaderboard number.
- Stage 1 (tune): per model, Optuna searches backend params (gpu_memory_utilization /
  max_model_len / batch, or n_gpu_layers + ctx for llama.cpp) and RAG params
  (chunk_size, overlap, top_k) over a TIGHTLY BOUNDED space, SMALL budget (~15-25 trials)
  against the FIXED, seeded tuning split. The embedding model is PINNED per Premise 4 and
  is NOT an Optuna parameter (searching it would reintroduce the retrieval confound).
  Objective = quality with does-it-fit-in-VRAM as a HARD constraint (over-VRAM configs
  pruned, not fatal) and latency as a tie-breaker.
- Stage 2 (score): run the FULL gold set once on the winning config. That run is the
  canonical leaderboard entry.
- Optuna trials obey the same hard-isolation + VRAM + thermal gate. The study uses
  persistent SQLite so crashed/OOM trials are resumable. Tune trials are nested MLflow
  child runs under the model's parent run; only the stage-2 run is tagged as the
  leaderboard entry.

### Expanded output
- Each leaderboard entry is now (model, best backend config, best RAG config) for THIS
  machine, scored at that best-config point. A model with no VRAM-feasible config is
  marked infeasible on this hardware.
- The thin Streamlit board surfaces the winning backend + RAG config alongside the rank,
  so the deliverable is "deploy THIS model with THIS config," not just a name.

### Rigor findings folded in
- `VramNotReclaimed` aborts the suite loudly; no silent bias.
- Eval-graph node failures (empty / malformed / refusal) are typed and scored as
  reliability failures, not crashes.
- MLflow logging writes scores to a parquet fallback FIRST, then MLflow, so a store
  error never loses a completed run.
- The Optuna proxy subset is fixed and seeded across all trials and all models.
- Corpus text is untrusted input to judge/RAG prompts: keep it in a delimited field. If
  the judge is an external API, corpus egress is a deliberate decision (Open Question 2).
- Harness unit tests: VRAM gate (mock NVML), backend resolver priority, recall@k/MRR
  fixtures, Optuna constraint-pruning on an over-VRAM config, resume-after-kill; plus one
  CUDA-free end-to-end smoke on a tiny Ollama model.

### Honest effort update
These additions (two-stage Optuna, LangGraph for all eval flows, MLflow integration,
thermal-gated isolation) push realistic v1 from ~1-2 weeks to ~2-3 weeks of human
effort, still gold-set-dominated. The trade buys reproducible, unbiased, per-machine-
optimal results, which is the point of "production-grade."

### Outside-voice resolutions (Codex, 2026-06-19)

A second Codex cold read challenged the new tooling. Decisions the user ruled on:
- Optuna: KEPT for all models, methodology fixed (disjoint gold-set partitions, embedding
  pinned, tightly bounded search space, Pareto-first output).
- LangGraph: REAFFIRMED for all eval flows (uniform pattern + clean path to the deferred
  agentic phase, accepting per-call overhead).
- Reproducibility: canonical immutable manifest (JSON/YAML) + Parquet are the source of
  truth; MLflow mirrors for its UI.
- Thermal: CAPPED configurable cooldown (wait up to a max, then proceed and flag
  thermally-elevated) + always record temp/clocks/power, not an unbounded hard gate.

Folded-in refinements (no decision needed):
- Output is "best OBSERVED config under this run environment" (driver/backend versions +
  thermal state recorded), not a durable per-machine optimum.
- tokens/sec measured at STEADY STATE after warmup; cold-start model-load time recorded
  as its own metric (don't conflate cold-start purity with steady-state throughput).
- Failure taxonomy is DISTINCT typed cases (empty / malformed / refusal / timeout /
  context-truncation / retrieval-miss / backend-crash / OOM / judge-failure), each
  recorded separately, not collapsed into one "reliability failure."
- Ragas is treated as a component to VALIDATE (Ukrainian metric-prompt localization,
  judge behavior, reference scoring), not settled infrastructure (see Open Question 2).
- vLLM params enter Optuna ONLY for candidates whose resolved backend is vLLM; the loop
  still proves out CUDA-free on Ollama first.

### Data-prep utilities (scope added 2026-06-19)

Two generation utilities added to v1 to lower the gold-set / curation barrier (the #1
risk). They call frontier cloud APIs (gemini / openai / anthropic), NOT local backends,
so they run OUTSIDE the sequential GPU isolation path. This partially un-defers
"automated dataset generation" from Premise 6, on purpose.

- Frontier client: one lightweight multi-provider interface (litellm) for
  gemini/openai/anthropic. API keys via env/.env per AGENTS.md (no hardcoding). Per-call
  cost + provider + model recorded.

1. `prepare-goldset` (RAG golden-set prep). Generates DRAFT (question + candidate source
   chunk + reference answer) triples from the user's corpus (wrapping Ragas's testset
   generator where it fits). Output is DRAFT for human review/edit; nothing enters the
   gold set untrusted. Semi-automates Premise 3 WITHOUT breaking the "defensible"
   premise: a human still verifies every item that scores models. [eng-review default,
   confirm in Architecture]
2. `prepare-synthetic-corpus` (synthetic narrative/topic corpora). Generates test
   documents / chat logs with KNOWN planted narratives, hidden topics, trends, and
   problems, recording the planted ground-truth as labels. Gives objective ground truth
   for the text-analysis / chat-period-analysis benchmark (trend / insight / topic
   recognition) without hand-labeling. Constraints: (a) plant topics/narratives/trends
   as STRUCTURED labels recorded with the text, so recovery is scored OBJECTIVELY where
   possible (did the candidate name planted topic T, surface planted trend X) with no
   LLM judge; (b) for any free-form quality that still needs judging, the planting
   provider MUST differ from the judge provider (e.g. plant with gemini, judge with
   anthropic) to avoid self-grading circularity; (c) data is synthetic, so no real-corpus
   egress, but TAG it synthetic in the manifest so it is never conflated with real-corpus
   results.

Both are "data prep" commands feeding the gold set / test corpora; they do not touch the
VRAM/thermal isolation path (no local GPU). They unlock the text-analysis benchmark with
objective ground truth.

### Eng-review resolutions (2026-06-19)

Engineering-readiness pass + a full Codex outside-voice. Build sequencing: WALKING
SKELETON first (one Ollama model, fixed config, hand-built seed gold set, pinned

Decisions ruled on (correctness fixes from Codex):
- Ranking axis: generation models ranked by REFERENCE-BASED ANSWER CORRECTNESS (+ gated
  judge), NOT retrieval recall (recall@k/MRR is constant across models under pinned
  retrieval; it validates the embedding only). Judge-demoted fallback = reference answer
  correctness.
- Gold retrieval labels are SOURCE-SPAN (document id + char offsets), not chunk IDs; a hit

  chunk-policy/label circularity.
- Gold set grows to ~200-300 RAG items (drafted via prepare-goldset, EVERY item human-
  verified) so disjoint calibration/tuning/final splits each have power. Judge calibration
  uses ~50-80 rated answers with quality diversity including adversarial fluent-wrong
  examples. Every leaderboard number and the judge rho carry a CONFIDENCE INTERVAL.
  (Supersedes the 50-150 / 30-answer figures in Premises 2-3.)

Folded-in refinements (Codex, no decision needed):
- Fixed throughput protocol: fixed prompt set + fixed max_new_tokens + N warmup iters,
  measure steady state; record prompt/gen lengths so tokens/sec is comparable.
- Synthetic-corpus results scored and reported SEPARATELY from real-corpus results, never
  merged into one headline number. Benchmark limitations + uncertainty bands are first-class.
- Text-analysis scoring SCHEMA (what counts as recovering a trend/topic/narrative) is an
  explicit open question to define before that benchmark is trusted; structured planted
  labels make the checkable part objective.
- The OpenAI-compatible adapter normalizes per-backend correctness quirks (stop sequences,
  context truncation, token accounting, streaming, error semantics), not just telemetry.
- Resolve the first concrete candidate-model list (Open Question 3) BEFORE finalizing
  launcher priority / install path / quant assumptions / vLLM timing.
- Walking-skeleton milestone 2 adds one REAL backend + telemetry path (not only CUDA-free
  plumbing) so CUDA / HF-loading / tokenizer assumptions are validated before the full sweep.
- Prep-util provenance: every gold/corpus item records provenance (frontier-drafted /
  human-authored / human-verified); only human-verified items score models.

## Prior-Art Integration (lang-uk / INSAIT / MamayLM, 2026-06-19)

Studied three Ukrainian-eval efforts and embedded their experience. Net: they REINFORCE
the wedge (public rankings don't transfer; even lang-uk's board mixes 0-shot and 3-shot
and is internally non-comparable), so we consume them as a free prior + ready-made data
and keep the private corpus eval as the decider.

Decisions ruled on:
- TWO-TIER eval. Tier 1 = a PUBLIC SCREEN via INSAIT's `lm-evaluation-harness-uk`
  (EleutherAI fork; already supports vLLM + OpenAI-compatible local-completions) running
  standard UA tasks (ZNO, MMLU-uk, Belebele-uk, SQuAD-uk, IFEval-uk, GSM8K-uk) to narrow
  the candidate field cheaply (no gold set needed). Tier 2 = loc-lm-bench's private
  corpus-grounded RAG + text-analysis eval + Optuna, run ONLY on the screen's survivors.
  The screen operationalizes "tune finalists only." New CLI: `screen-public`.
  - Screen EXECUTION (eng-review decision): lm-eval-harness runs in `local-completions`
    mode against loc-lm-bench's OWN launched OpenAI-compatible endpoint, so model launch +
    isolation stays under the executor (one launch path, not two). Caveat: log-prob MCQ
    tasks (ZNO etc.) need the server to expose logprobs, so they run only on logprob-capable
    backends (vLLM); Ollama-served candidates get the generative-task subset, or are
    screened on vLLM when HF weights exist. The screen adapter records, per candidate,
    which tasks ran vs were skipped for lack of logprobs, so the screen is never silently
    partial. Candidates are split into TWO SCREEN TRACKS, logprob-capable (vLLM: full MCQ
    + generative) and generation-only (Ollama), and are NEVER ranked across tracks (Codex
    flagged HTTP-loglikelihood brittleness and verbosity bias; the track split contains
    it). MCQ-over-HTTP fidelity is accepted as a coarse-filter risk, recorded per run.
- COMPOSITE = AVERAGE-RANK headline (scale-invariant, matches lang-uk, comparable to
  their public board) + the configurable weighted-blend as a secondary view + Pareto +
  CIs. Disagreements between the two views are surfaced, not hidden. Refinements (Codex):
  Tier-1 screen metrics and Tier-2 private metrics are NEVER combined in one rank (screen
  prunes, private selects); rank flips within overlapping CIs are marked "statistically
  unresolved"; the Pareto + per-metric CIs stay as prominent as the average-rank line;
  the weighted-blend view is labeled policy-dependent, not an objective leaderboard.
- DATASET REUSE. SQuAD-uk + Belebele-uk (public, span-labeled UA in-context QA, which
  lang-uk treats as a RAG proxy) seed and smoke-test the RAG loop in Milestone 1 before
  the hand-built corpus gold set exists, and serve as a transfer baseline.
- CANDIDATE MATRIX adds Ukrainian-specialized models: MamayLM v2 12B + 27B, Lapa LLM,
  Gemma 3 (resolves part of Open Question 3).

Folded-in refinements (no decision):
- Batch-invariance: enable vLLM batch-invariant execution so outputs are reproducible
  regardless of batch size. Directly serves the unbiased/reproducible requirement;
  lang-uk roadmaps the same.
- Record N-SHOT setting in every manifest and FIX it across compared models (the public
  board's 0-shot vs 3-shot mixing is a comparability bug we avoid).
- Add a tokenizer-efficiency metric (tokens per UA char/word); lang-uk roadmaps the same.
- ZNO-Eval answer parsing is fiddly (lang-uk flags it); budget for robust answer
  extraction in the screen.
- Judge candidate: MamayLM v2 27B is a plausible LOCAL Ukrainian judge (Open Question 2),
  still gated by the rho>=0.6 calibration; a frontier API remains the safer default.
- loc-lm-bench is NOT a public leaderboard; it consumes lang-uk/INSAIT results as a prior
  and does not duplicate them.

Sources: MamayLM v2 (models.mamay.ai/blog), INSAIT lm-evaluation-harness-uk (github.com/
insait-institute), lang-uk Ukrainian LLM Leaderboard (huggingface.co/spaces/lang-uk) +
its results dataset (lang-uk/ukrainian-llm-leaderboard-results).

### B. Planning formulas and tuning tables (Approach B / deferred)

- effective_context = min(model_max_context, backend_supported_context,
  hardware_supported_context, benchmark_required_context).
- Total VRAM ~= model_weights + KV_cache + backend_overhead + CUDA_runtime_overhead.
- KV-cache relative scaling vs 8K baseline: 8K=1x, 16K=2x, 32K=4x, 64K=8x, 128K=16x.
- vLLM gpu_memory_utilization by VRAM: 12GB=0.80, 16GB=0.85, 24GB=0.90, 48GB+=0.92.
  On KV-cache overflow: reduce max_model_len, retry, record requested vs served
  context + reason.
- Ollama: num_ctx auto, num_gpu auto; pick safe num_ctx from VRAM + model size + KV
  estimate; record requested vs selected context.
- llama.cpp: n_gpu_layers / ctx_size / batch_size auto; maximize context while
  keeping inference stable.
- Hardware classes: A (12GB: 3-8B, Q4/Q5, <=16-32K), B (16GB: 7-14B, Q4/Q5/Q8,
  <=32-64K, the primary target), C (24GB: 14-32B quant or 7-14B FP16, <=64-128K),
  D (48GB+: 32-70B quant, large-context / long-doc / agentic).
- Architectures to recognize: Ampere, Ada Lovelace, Hopper, Blackwell, plus
  workstation/datacenter variants. Parse and preserve raw nvidia-smi output (driver,
  CUDA, SMI version).

### D. Full benchmark taxonomy (the "later" scope, recorded precisely)

v1 implements only Text Analysis + RAG. The original full suite and default weights,
for when the framework grows:

- Text Analysis 20% (key-fact / entity extraction, topic / trend / narrative
  recognition, risk / decision extraction, contradiction detection, insight
  generation, long-doc comprehension).
- Summarization 10%, Structured Output 10%, RAG Quality 20%, Chat-Period Analysis
  10%, Security 10%, Agentic Workflows 10%, MCP and Tooling 5%, Reliability 5%.
- Full composite (later): final = quality*0.60 + reliability*0.15 + security*0.10 +
  agentic*0.05 + tooling*0.05 + efficiency*0.05, all components normalized 0-100.
- Agentic frameworks (deferred): LangGraph (primary), LangChain, LlamaIndex,
  Haystack, CrewAI, AutoGen.
- Security suite (deferred): prompt-injection, jailbreak, instruction-hierarchy,
  unsafe-content, tool-abuse, RAG-injection, data-exfiltration resistance.
- Full per-run output set (target): run_manifest.yaml, scores.parquet,
  telemetry.parquet, retrievals.parquet, responses.jsonl, leaderboard.sqlite,
  leaderboard.duckdb. Leaderboard tables: runs, metrics, models, hardware, backends,
  leaderboard_entries; indexed on model, backend, hardware_profile, timestamp, score.
- Utilities envisioned (original idea): (1) a task builder specifying models +
  inference params that also offers alternative models for the hardware; (2) a
  RAG-variant generator that builds several vector-base variants and checks
  chat-format compliance; (3) a sequential test runner recording results + metadata
  + system prompts. v1 collapses these into the Typer CLI commands.
